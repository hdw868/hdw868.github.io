Recommended Material to Learn Cucumber[Beginner]

1.Learning materials

When learning cucumber I followed a popular book named “The cucumber book”, which has plenty vivid examples through the whole book.

Thanks to the safaribook-online program again, it’s also available here:

https://www.safaribooksonline.com/library/view/the-cucumber-book/9781680502497/

You don’t have to go through all the book I suggest you take a look of Part-I and then go through Chapter 12. Testing a REST Web Service to get a quick start.

2.Questions

Basically, cucumber gives us a guidance of how we should develop our product with BDD.

At the mean time, it also shows us how we should describe our requirement(can be the test cases as well)

When writing our test cases, I suggest we have an understanding of the following concept, which will help us organize our test cases to better describe what exactly we want to test:

  • Concept of Feature, Scenario, Given/When/Then

  • Use Background to specify a set of steps that are common to every scenario in the file;

  • Use Scenario Outline and Examples Tables to collapse repetitive scenarios;

  • Hooks and Tags: Before, After, Around, @tag

  • Command-Line options:

    $cucumber –tags @focus，@acceptance

    $ cucumber features/something.feature:45

    $ cucumber –name logout

    $ cucumber -f pretty -f html –out result.html

    $ cucumber myfeature

    #in fact, every time you run “cucumber”, it always add the “features” parameter automatically

3.Armed yourself with the latest feature of cucumber 3.0.0

The cucumber book was wrote a few years ago, the latest cucumber version is now 3.0.1 when I write this article, I find the following new features of cucumber that is pretty cool:

It relief you from writing the complex regexp with more readability simple express!

1.Transform is no longer with Cucumber. Welcome its replacement - ParameterType.

On the surface, ParameterType is similar to Transform - it lets you transform an argument from a string to another type before it’s passed to the step definition.

Prior to 3.0.0 you might have defined a Transform for integers:

In 3.0.0, int conversion is built-in, so there is no need for that transform.

You can also define your custom type, see more details from the official blog.

2.Regular Expressions to Cucumber Expressions

As shown above, you can keep using regular expressions and you can leave them untouched unless you’ve been using them with transforms, or you’ve got nested capture groups.

Here are some tips about how to translate Regular Expressions to Cucumber Expressions:

Anchors

The gorgeous ^ and $ you love so much are no longer needed. Just get rid of them.

Capture Groups

All capture groups become Cucumber Expression output parameters. There are 4 built-in ones to choose from:

  • (\d+) becomes {int}
  • Whatever funky pattern you’ve been using for floats becomes {float}
  • (\w+) becomes {word} (a word without quotes)
  • "([^"]+)" becomes {string} (anything between single or double quotes)

测试框架开发小结(Draft):

一个好的测试框架应该具备以下特征:

Basic:

  • 准确, 测试结果必须是可靠的, pass就是pass, fail不是因为runtime error;

  • 稳定, 单个测试的fail不影响其他case的执行, 但某个错误会导致后续执行无意义则应尽早退出;

  • 灵活, 测试执行的范围是可灵活配置的, 按特性, 按tag, retry功能; 测试数据的允许自定义, 允许用户执行测试前执行特定的脚本, 允许用户配置测试执行的环境变量(比如token, key, username, pwd等信息);

  • 易拓展, 用户如何新增自己的测试场景, 如何定义自己的特定配置;

  • 完整性, 下载安装一次到位, 不需要多次安装依赖包等操作;

    Nice to Have:

  • 并发执行, 测试执行可以再多个环境上并发执行;

  • 兼容性, 支持跨平台执行;

  • 有数据分析, 一个好的测试框架应该提供直观的测试数据分析功能, 包括测试通过率, 执行时间, fail track, sync with bug system, write result to db;

需要明确的地方:

明确测试流程, 测试依赖, 测试数据来源, 测试干扰(网络延迟)

实现层面的考虑:

  1. Exception Hierarchy设计, custom Exception, handle Exception;
  2. logging系统设计, console输出, file输出;

  3. Test Object设计, feature, scenario, case

    • 抽象类Test Object, tag, dependency, configuration属性, run_test, setup, teardown方法;

    • feature包含scenario, scenario包含case;

  4. Report系统, 将测试数据结构, pass, fail, skip数据以html方式展示;
  5. DB系统, 测试结果回写到后台数据库;
  6. Unix和Windows平台兼容性, 分隔符, 换行符处理等;
  7. 多线程, 测试数据准备, 测试执行并发系统(队列系统);
  8. 任务分发系统, 测试分发到不同executor上执行;

Python层面:

  1. 谨慎的使用tuple作为返回值, 有些返回值使用字典更加灵活;
  2. 设计好目录层级, 参考cucumber的folder;
  3. 测试的执行可以用配置文件来管理, 提供默认配置和按指定配置执行;

Python开发利器-PyCharm快速上手

一开始接触PyCharm时, 其实我是, 拒绝的. 早期为了打好基础一直坚持用Atom做开发, 日积月累习惯了便习以为然. Atom有诸多优点, 丰富的插件, 开源免费, 轻量级; 当然最大缺点还是性能问题, 打开一个10M左右的文本肯定会停止响应导致只能强行关闭了. 另外,缺乏go to definition(有插件,但是支持比较差, 经常碰到无法跳转的情况)实在很不方便. 我想是时候试试主流IDE了, 然后当我试了一下PyCharm, 发现”duang~”, 真的好用.

Note: 这里建议大家在学习初期还是坚持不要太依赖IDE提供的编辑功能, 这些功能太便捷了以至于有时离开ide出错之后你错的莫名奇妙. 我见过不少同学依赖IDE把当前项目路径加到path, 导致这种脚本经常换个环境就嗝屁.

扯远了, 下面给出干货, 基于使用场景, 整理了若干技巧, 这些技巧只要基于场景有意识的使用就能迅速掌握, 不需要死记硬背. 这些都是我个人极力推荐熟练使用的功能, 帮你写代码时行云流水, 写完后神清气爽~

  1. 全局搜索: Ctrl+Shift+F,

    不过PyCharm的更强大, 你可以点选左侧某个目录后再按Ctrl+Shift+F, 这样默认会搜索改目录;

    tip: 如果只是想高亮下文件, 只要选中Project Navigator然后直接打字就会自动高亮所有匹配的文件;

  2. 添加注释: Ctrl+/, 再按一遍取消注释

  3. 重构Refactor: Ctrl+Alt+Shift+T

    有时候代码写了好长了, 发现当初文件命名不合理, 手动修改需要遍历所有引用的地方? 这时选择对应的文件/文件夹后, Ctrl+Alt+Shift+T后选择 rename; 如果是文件目录分类不合理希望移动就选move; 如果选择代码中的任意变量或者常量, function name等等, 选择rename, 自动帮你把所有引用的地方和定义修改掉, 而且提供preview功能确保不会错误的修改某些地方.

  4. 自动PEP8风格: Ctrl+Alt+L

    满眼的波浪线提示你代码风格混乱时用一下, 世界清净了;

  5. 智能纠正: Alt+Enter

    在还没import(install)模块名上使用会import(install)相对应的模块，免去跑到代码顶部写import;

    在方法名上使用补注释，在测试中使用帮你补充断言代码……不管出错了不知道怎么办，知道怎么办想这么办时都按下这个快捷键，有惊喜!

  6. 运行当前文件: Ctrl+Shift+F10

    十分智能，如果是测试代码，则以测试模式run，并列出测试通过情况

  7. 查注释(docstring): Ctrl+Q

  8. 查定义的(源代码): Ctrl+B/Ctrl+I

  9. 查看参数: Ctrl+P

    在括号内按下，就知道当前位置该给啥参数

  10. 另起一行: Shift+Enter

    光标在行内任意位置都能另起一行，且不破坏当行结构;

  11. 复制当前行: Ctrl+D

  12. 删除当前行: Ctrl+X

  13. 弹出常用的snippet: *Ctrl+J

    比如: if name == “main“:

  14. 搜索一切: 双击shift

    搜索一切，不管是IDE功能、文件、方法、变量……都能搜索

  15. Ctrl+光标

    当按住ctrl，鼠标移到标识符上面时，会显示一些提示信息。

    当按住ctrl，并点击时，可以跳到定义处或者弹框列出所有引用的地方，只有一个地方引用时，会直接跳

  16. 最近打开的文件: Ctrl+E

  17. 智能补全Tab:

    这个应该都知道, 补充一个, 输入a.print然后按tab有惊喜

  18. Surround with: Ctrl+Alt+T

    比如将光标所在的语句或块用try…except…包住。

  19. Extend Selection: Ctrl+W

    往外扩展选择范围。我一般用来选中单词、两个引号或括号之间的内容。

  20. Surround selection on typing quote or brace:

    这个是个setting, 设置一下方便不少, Ctrl+Shift+A搜索smart keys, 勾选Surround selection on typing quote or brace, 这样选中文字后引号或者括号或自动加到头尾两侧, 而不是替换文字;

  21. Debug, VCS功能, 这几块也是很强大的, 讲解起来比较费劲, 强烈建议大家看下官方的介绍视频, 受益匪浅!

如何养成跑步的习惯

背景

2016年4月出差回来之后，由于意识到自己的体重在毕业以后从58一路狂奔到72，长时间的电脑办公滋生了颈椎病。

那段时间正好在备孕，看着手机上今昔照片的对比，暗暗下定决心要开始锻炼身体了。

关于锻炼方式，因为我从小没有运动天赋，球类运动需要特定场馆而且需要有个固定的同伴比较好。

而我对于去健身房锻炼一向持保留态度（主要是穷），我一直坚信只要有心，在哪里锻炼都是可以的，所以剩下的只有一个选择了——跑步。

优点显而易见：

  • 无门槛（谁还不会跑步啊？！适合我这种没啥运动天赋的人）；
  • 自由（一个人想跑就跑，不像球类运行需要互相协调时间和场地）；
  • 有氧运动（我不会游泳，所以减肥最好的有氧运动就变成跑步了）；
  • 经济（你唯一必须的投资就是一双跑鞋，一套速干衣）

缺点嘛：

一个是枯燥，我跑步的时候不听歌，一个人跑40多分钟，黑灯瞎火的也没有啥靓丽的风景，如果不找点事确实是比较枯燥的。所幸的是，跑步的过程中脑子特别清醒，我利用这段时间将自己从一天喧嚣的工作中解脱出来，彻底放松自己，用来思考一些问题。

第二个就是计划赶不上变化，户外跑是很看天气的，杭州的梅雨季节我曾经停了3个星期没有去跑步。所以要提前做好规划，为了达到一周锻炼3-4次的要求，我买了个瑜伽垫，下雨天没法锻炼的时间就用keep补运动量。

过程

由于常年没有运动，一开始我只跑3.5公里。

那段时间家庭出现了重大经济危机和矛盾，工作上总有处理不完的棘手问题等着你，于是跑步成了我一天唯一的期盼。在我不知道应该如何应对眼前的问题的时候，跑步成了我唯一的坚持。虽然每次跑完都很累，但是比跑步更痛苦的是眼前的一堆问题。跑步成了我能短暂逃离现实的一种放松方式，同时也帮我在生活工作中慢慢的重拾信心。通过跑步，我为自己的意志力和信心充值。那段时间我正好在看《富爸爸穷爸爸》和《自控力》这两本书，从中收获良多并且结合自身情况也有诸多思考。跑步过程有个著名的“假疲劳”现象，就是你的疲惫感更多的是大脑自我保护的一个反应，并非真实的身体极限。当我将注意力集中在如何达成目标，而不是未自己寻找失败的原因的时候，一切变得不是那么困难了。我经常告诉自己，跑过这座桥才能停下，过了这个路灯就要开始跑步，不能在终点前停下。当我习惯于按照自己设立的原则和目标行动时，我确实跑的更远。我当时给自己立的标准是每个月的单次跑步里程必须增加500米，就这样我从开始的3.5公里，到现在的稳定7公里用了半年多时间。

当然，我想对于初学者来说，绝大多数人不是死在最后的1公里，而可能是开始的那100米。

就我自身而言，我发现如果我没有完成目标，极少是由于体力不支没有跑完。而90%的情况下，我是死在第一步——换衣服。因为到家时间比较晚，我一般要在8点以后才会拥有可自由支配时间，由于多年沉溺游戏的恶习，我经常一屁股就坐到电脑前面开始所谓的“relax time”。等到8点30我的streaks应用提醒我该运动时，一想到跑步这个漫长枯燥的过程，人的惰性渐渐战胜了理性，然后就这样平白无故缺勤了。这种状态一直持续了很久，我从开始的每周4跑渐渐的变成每周2跑，打卡多次中断。不断的反思和摸索中，我终于找到一种解决办法，我不再思考跑完需要花费我45分钟的时间这件事，而是聚焦于迈出第一步。比如今天该跑步了，到点的时候我只想着先去换个衣服吧，换完衣服时就自然而然去热个身，等热身完了我就下楼了，然后到了户外，看着自己自身行头跟前面的热身，想着不跑都对不起自己啊，于是就这样上路了！

通过分解任务，我将跑步这个过程划分为：

更换衣服》热身》下楼》迈开步子》跑完既定公里数

如果你也有同样的问题，不妨试试先分解任务。简单的将问题归因于自身意志力不足实际上是思想上的懒惰!

后记:

-写这篇文章是在2017年4月, 当时每次最远距离大概在8公里左右, 配速在每公里6分15秒左右;

-2018年5月5日更新, 目前每周大概跑步三次, 累计一个月的跑量在100-120公里左右. 最远距离在15公里, 10km的配速在每公里5分15秒左右;

回首这一年来的进步, 主要有两点, 一是偶然报名小马拉松时正好进了一个马拉松群, 各路大神每次打卡都是10km以上的, 在这种氛围刺激下不自觉的也提高了对自己的要求; 二是想着既然花那么多时间跑步了, 为什么不多努力一点让自己跑的更好更远呢? 于是开始系统的学习跑步的知识, 现在跑步已经完全融入我的生活, 并且一个好习惯常常会带来连锁反应, 早睡早起, 看书跑步, 写作思考, 这些习惯也能养成!
